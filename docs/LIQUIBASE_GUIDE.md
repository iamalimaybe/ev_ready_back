# Liquibase Guide

Keep the existing XML master changelog if it only acts as an `includeAll` wrapper.

Do not replace the XML master changelog with a SQL master changelog unless explicitly asked later.

Actual schema and seed changes must be written as SQL formatted Liquibase files.

The current master changelog should remain:

```text
src/main/resources/db/db.changelog-master.xml
```

`application.yml` should continue pointing to:

```text
classpath:/db/db.changelog-master.xml
```

Future SQL files should live under:

```text
src/main/resources/db/changelog/
```

Naming pattern:

```text
V001__create_vehicle_table.sql
V002__seed_vehicle_data.sql
```

Changeset IDs should include the same V-prefixed number as the filename:

```text
--changeset evready:V001-create-vehicle-table
```

For files with multiple changesets, every changeset ID should start with the file's V-number prefix.

## Formatted SQL Pattern

```sql
--liquibase formatted sql

--changeset evready:V001-create-vehicle-table
--preconditions onFail:MARK_RAN onError:HALT
--precondition-sql-check expectedResult:0 SELECT COUNT(*) FROM information_schema.tables WHERE table_schema = 'public' AND table_name = 'vehicle';
CREATE TABLE vehicle (
    id BIGINT GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
    name VARCHAR(150) NOT NULL,
    created_at TIMESTAMP WITH TIME ZONE NOT NULL DEFAULT CURRENT_TIMESTAMP,
    created_by VARCHAR(100),
    updated_at TIMESTAMP WITH TIME ZONE,
    updated_by VARCHAR(100)
);
```

## Migration Rules

- Use preconditions for create table scripts.
- Use `onFail:MARK_RAN` when the table already exists.
- Use `onError:HALT` for real SQL/precondition errors.
- Use `BIGINT` primary keys, not UUID.
- Include audit columns in every table unless explicitly told otherwise.
- Do not use YAML or XML changelog files for actual migrations.
- Keep changeset IDs aligned with the filename's V-number prefix, for example `evready:V004-create-vehicle-table`.
